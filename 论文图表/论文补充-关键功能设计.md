# 4.3 关键功能设计

本节对系统的五大关键功能进行详细设计说明，包括功能描述、处理流程和关键设计决策。

## 4.3.1 JWT 用户认证流程

用户认证是系统的安全入口，采用 JSON Web Token（JWT）无状态认证机制，结合 bcrypt 密码哈希算法保障用户凭证安全。

**认证流程：**
```
客户端                          服务端
  │                               │
  ├─ POST /api/auth/login ───────>│  发送用户名、密码
  │                               ├─ 查询 users 表验证用户名
  │                               ├─ bcrypt.verify() 比对密码哈希
  │                               ├─ 失败 → 返回 401 "用户名或密码错误"
  │                               ├─ 成功 → 生成 JWT:
  │                               │   Header:  {"alg":"HS256","typ":"JWT"}
  │                               │   Payload: {"user_id":N,"username":"X",
  │                               │             "exp": now+7天}
  │                               │   Signature: HMAC-SHA256(header.payload, SECRET_KEY)
  │<────── {access_token, user} ──┤
  │                               │
  ├─ GET /api/auth/me ───────────>│  携带 Authorization: Bearer <token>
  ├─ Header: Authorization ──────>│
  │                               ├─ 解析 Bearer Token
  │                               ├─ jwt.decode(token, SECRET_KEY, algorithms=[HS256])
  │                               ├─ 提取 user_id → 查询 users 表
  │<────── {id, username, ...} ───┤
```

**关键设计决策：**
- Token 有效期设置为 7 天，在安全性与用户体验间取得平衡
- 密码采用 bcrypt 算法哈希存储，salt 轮数为 12，即使数据库泄露攻击者也难以还原明文
- `require_auth` 依赖注入：路由函数声明 `Depends(require_auth)` 即可强制要求登录态，未登录自动返回 401

## 4.3.2 视频浏览与多维度检索

视频浏览是系统的核心交互入口，用户可在首页查看视频列表，并通过关键词搜索、分类筛选、多维度排序和分页浏览实现灵活的数据探索。系统当前存储 656 条来自 B 站热门 API 的真实 ACG 视频数据，涵盖 12 个内容分类。

**检索流程：**
```
用户输入 → 前端收集参数 → GET /api/videos?keyword=&category=&sort_by=&page=&page_size=
                                    │
                    后端 SQLAlchemy 查询构造器
                    ┌───────────────────────┐
                    │ query = db.query(Video)│
                    │ if keyword:            │
                    │   query = query.filter( │
                    │     Video.title.ilike( │
                    │       f"%{keyword}%")) │
                    │ if category:           │
                    │   query = query.filter( │
                    │     Video.category==x) │
                    │ if sort_by==play_count:│
                    │   query.order_by(desc( │
                    │     Video.play_count)) │
                    │ total = query.count()  │
                    │ videos = query.offset( │
                    │   (page-1)*ps).limit() │
                    └───────────────────────┘
                                    │
                        返回 {items, total, page, page_size}
```

**支持的排序维度：**
| 排序字段 | 说明 | 典型用途 |
|---------|------|---------|
| play_count | 按播放量降序 | 查看最热门视频 |
| like_count | 按点赞数降序 | 查看最受欢迎视频 |
| danmaku_count | 按弹幕数降序 | 查看讨论最活跃视频 |
| publish_date | 按发布时间降序 | 查看最新发布视频 |

**前端分页设计：**
- 每页默认 20 条，支持滑动窗口导航
- 页码显示策略：始终显示首页、尾页，当前页前后各 3 页，其余以省略号折叠
- 上一页/下一页按钮，当前为首页/尾页时自动禁用

## 4.3.3 数据看板聚合分析

数据看板是面向数据分析人员的深度分析模块，提供 7 个维度的统计图表。与可视化大屏的"一屏总览"不同，数据看板强调交互式探索和多维度交叉对比。

**核心分析指标设计：**

（1）**互动评分公式**：为综合评估各类别视频的用户互动水平，设计加权评分模型：
```
engagement_score = like_rate × 0.35     // 点赞率权重 35%
                 + danmaku_rate × 0.20   // 弹幕率权重 20%
                 + reply_rate × 0.20     // 评论率权重 20%
                 + coin_rate × 0.15      // 投币率权重 15%
                 + share_rate × 0.10     // 分享率权重 10%

其中各项 rate = SUM(指标) / SUM(播放量) × 100%
```
权重分配依据：点赞和弹幕是 B 站 ACG 社区最核心的互动形式，合计占 55%；评论和投币反映深度参与，合计占 35%；分享反映传播意愿，占 10%。

（2）**时长区间分桶**：将视频按时长划分为五个区间：
```
<1分钟    (0-60s)    — 短视频 / 动态内容
1-3分钟   (60-180s)  — 碎片化内容
3-10分钟  (180-600s) — 常规视频
10-30分钟 (600-1800s)— 深度内容
>30分钟   (>1800s)   — 长视频 / 录播
```

（3）**30 天趋势**：基于 `video_stats_history` 表按时序聚合，以日期为横轴、播放/点赞/弹幕数三线为纵轴，直观反映数据随时间的变化趋势。

**后端数据聚合模式：**
```
SQLAlchemy 聚合查询 → GROUP BY category → SUM(play_count), SUM(like_count), ...
    → Python 后处理: safe_rate() 防除零 → 计算 weighted score
    → 返回 JSON → 前端 ECharts 渲染
```

## 4.3.4 可视化大屏设计

可视化大屏采用暗色科技主题（深蓝黑背景 #0f0f23，粉 #f472b6 / 青 #00d4ff / 紫 #a78bfa 为主色调），整体布局为三栏式结构：

```
┌─────────────────────────────────────────────┐
│  实时时钟           标题           全屏按钮  │
├────────┬──────────────────┬─────────────────┤
│ KPI卡片 │  TOP20 视频排行   │  弹幕活跃度排行  │
│ (4项)  │  (横向渐变柱状图)  │  (蓝绿渐变柱状图) │
├────────┤                  ├─────────────────┤
│分类分布 │                  │  UP主影响力排行   │
│玫瑰图  │                  │  (金橙渐变柱状图) │
├────────┼──────────────────┼─────────────────┤
│热门标签 │  30天趋势面积图    │  互动6维雷达图   │
│TOP12   │  (3线渐变填充)    │  (多边形雷达)    │
└────────┴──────────────────┴─────────────────┘
```

**全屏模式设计：**
- 调用浏览器原生 Fullscreen API：`element.requestFullscreen()`
- 兼容 Chrome（`requestFullscreen`）、Safari（`webkitRequestFullscreen`）、IE（`msRequestFullscreen`）
- 监听 `fullscreenchange` 事件自动更新按钮文案和图例标注
- 全屏切换后延迟 200ms 调用 `echarts.resize()` 重绘所有图表实例，适配新视口尺寸

**实时时钟设计：**
- 使用 `setInterval(fn, 1000)` 每秒刷新显示
- 格式：`YYYY-MM-DD 星期X HH:MM:SS`
- 页面切换离开大屏时调用 `clearInterval()` 清除定时器，防止内存泄漏

**ECharts 实例生命周期管理：**
- 全局维护 `charts[]` 数组，记录所有已创建的 ECharts 实例
- 页面导航切换时调用 `disposeAll()` 遍历数组逐一销毁
- `window.addEventListener('resize', ...)` 监听窗口尺寸变化，自动重绘

## 4.3.5 用户收藏切换

收藏功能采用"幂等切换"设计——同一接口同一视频反复请求即在收藏与取消收藏之间切换，简化了前后端交互逻辑。

**数据库交互流程：**
```
POST /api/videos/{video_id}/favorite  (需 JWT 认证)
    │
    ├─ 查询 user_favorites 表:
    │   SELECT * FROM user_favorites
    │   WHERE user_id=? AND video_id=?
    │
    ├─ 存在记录 (已收藏) → DELETE → 返回 {"favorited": false}
    │
    └─ 不存在记录 (未收藏) → INSERT → 返回 {"favorited": true}
```

**数据模型：**
- `user_favorites` 表为关联表，仅包含 `id`、`user_id`、`video_id`、`created_at` 四个字段
- `user_id` 和 `video_id` 均建立索引，加速查询
- 多对多关系：一个用户可收藏多个视频，一个视频可被多个用户收藏

**前端交互：**
- 视频详情页收藏按钮根据 `vdetail.is_favorited` 状态动态显示"⭐ 已收藏"或"☆ 收藏"
- 收藏列表页通过 JOIN 查询一次性获取用户所有收藏视频的完整信息
- 未登录用户点击收藏时弹出"请先登录"提示，不发送无效请求

## 4.3.6 评论系统 — 嵌套回复

评论系统支持用户对视频发表评论，并支持嵌套回复（楼中楼）结构，通过 `comments` 表的 `parent_id` 自引用外键实现树形评论结构。

**数据模型设计：**
```
comments 表:
  id         INTEGER PK     评论ID
  user_id    INTEGER FK     评论者 → users.id
  video_id   INTEGER FK     所属视频 → videos.id
  content    TEXT           评论内容
  parent_id  INTEGER FK     父评论ID → comments.id (NULL=顶层评论)
  created_at DATETIME       评论时间
```

**查询逻辑：**
- **顶层评论**：`SELECT * FROM comments WHERE video_id=? AND parent_id IS NULL ORDER BY created_at DESC`
- **子回复**：`SELECT * FROM comments WHERE parent_id=?` 查询指定评论的所有回复

**关键设计决策：**
- 仅支持一层嵌套（顶层评论 + 子回复），不实现无限层级的树形嵌套，降低查询复杂度和前端渲染开销
- 评论按时间倒序排列，最新评论优先展示
- 评论发布后即时刷新列表，使用户感知操作成功

## 4.3.7 本章小结

本节对系统的六项关键功能进行了详细设计说明：JWT 认证流程采用无状态 Token 机制保障安全；视频检索支持多维度组合查询与滑动窗口分页；数据看板设计了加权互动评分模型和时长分桶策略；可视化大屏采用三栏布局、全屏 API 和 ECharts 实例生命周期管理；收藏功能以幂等切换简化交互；评论系统通过自引用外键实现嵌套回复。各功能模块均已完成详细设计，关键接口和数据流已明确，为第五章的系统实现奠定了完整的技术基础。
