[P55|Heading 2] 1.1项目背景及意义
[P56|Normal] 随着数字娱乐产业的蓬勃发展以及二次元文化的广泛传播，哔哩哔哩（B站）已从最初的ACG（动画、漫画、游戏）内容创作社区，成长为国内最具影响力的视频分享平台之一。截至目前，B站拥有庞大的用户群体和海量的UP主资源，每天生产的视频内容数以万计，涵盖动画、游戏、音乐、舞蹈、生活等多个垂直领域。每一部视频背后都产生了包括播放量、点赞数、投币量、收藏数、弹幕评论等多维度的用户互动数据。这些数据不仅是衡量视频热度的直观指标，更蕴含着用户偏好变化、内容流行趋势演变及创作者生态健康状况的关键信息。然而，目前面向普通研究者、ACG爱好者及内容创作者的，能够系统化采集、深度分析并直观展示B站ACG视频数据的专用工具或平台仍较为匮乏。通用型数据分析平台如YouTube Data API或Social Blade，虽然提供了宏观的视频观看趋势和频道分析，但其关注点多为全球市场，对B站及国内ACG垂直社区的特色数据维度（如“一键三连”文化、弹幕互动模式等）缺乏深度适配与定制化分析能力。因此，本项目旨在设计并实现一个“B站ACG视频数据统计分析系统”。该系统通过自动化爬虫技术高效获取B站热门ACG视频的结构化数据，利用大数据处理工具进行数据清洗与整合，并基于Web技术构建交互式数据分析平台，实现多维度统计与可视化呈现。

[P58|Heading 3] 1.2.1 国内研究现状
[P59|Normal] 国内针对视频网站数据分析的研究近年来发展迅速，尤其以B站为代表的ACG社区成为热点研究对象。学者多聚焦于弹幕情感分析、UP主影响力评估及用户行为建模。例如，有研究利用LSTM模型对B站弹幕评论进行情感分类，有效识别用户情绪倾向[1]；也有学者基于B站高传播力视频数据，分析了播放量、点赞、投币等互动指标与视频特征之间的关联[2]。此外，Reaction视频中用户弹幕信息交互行为的情感反应生成机理也得到了深入探讨[3]。在爬虫技术与数据采集实践方面，基于Scrapy框架的电商数据分析与可视化研究为数据采集提供了成熟的技术参考[4]。数据清洗与标准化环节亦逐步受到重视。在用户行为分析与推荐算法领域，协同过滤、深度学习等方法已被系统综述并应用于视频推荐场景[5][6]。前端可视化方面，基于Vue的响应式数据可视化平台设计与实现以及ECharts数据可视化全攻略为构建直观的数据大屏提供了技术支持[7]。然而，现有研究普遍存在维度单一、实时性不足、缺乏交互式探索能力等问题，且将上述技术端到端地应用于B站ACG垂直领域、实现从自动采集到交互式分析完整闭环的系统化案例仍不多见。
[P60|Heading 3] 1.2.2 国外研究现状
[P61|Normal] 在视频数据分析领域，国外学者的研究起步较早，主要集中在以下几个方面，通用数据分析平台建设：以YouTube为代表的视频平台提供了官方数据接口（如YouTube Data API），支持开发者获取频道订阅量、视频观看趋势等宏观指标。第三方工具如Social Blade，进一步整合了多平台数据，提供社交媒体影响力分析。但这些平台主要面向全球市场，对特定文化社区（如ACG）的深度、定制化分析支持不足。视频流行度预测与用户行为分析：部分研究利用机器学习算法对视频传播规律进行建模。例如，部分研究利用长短期记忆网络对视频平台的观看趋势进行时序分析，取得了较好的预测效果。这些研究在算法层面取得了较好效果[8]，但往往停留在模型验证阶段，缺乏与工程化系统的集成。在开源数据采集与分析工具上国外开源社区存在大量基于Python（如Scrapy、BeautifulSoup）的网络爬虫项目，以及基于ECharts、D3.js的可视化库。

[P77|Heading 2] 2.1  Python
[P78|Normal] Python是一种解释型、面向对象、动态数据类型的高级程序设计语言，由荷兰程序员Guido van Rossum于1989年底开始设计，1991年发布第一个公开版本。Python的设计哲学强调代码的可读性与简洁性，其使用缩进来划分代码块，摒弃了传统语言中的大括号，使得程序结构清晰、风格统一。更为重要的是，Python的第三方库生态极为丰富，在数据采集领域有Requests、BeautifulSoup、Scrapy、Selenium等；在数据处理与分析领域有Pandas、NumPy、SciPy等；在Web开发领域有Django、Flask、FastAPI等；在本项目中，Python语言贯穿了系统的全部核心环节。第一，在数据采集层，本系统利用Python的Scrapy框架构建分布式爬虫，结合Requests库发送HTTP请求，使用BeautifulSoup库解析B站ACG视频页面的HTML结构，提取视频标题、UP主名称、播放量、点赞量、投币量、收藏量、发布时间、视频简介、分区标签等关键字段。同时，利用Python的随机模块和time模块实现User Agent轮换、请求延时等反爬策略，确保数据采集的稳定性与合规性。

[P80|Heading 2] 2.3 MySQL
[P81|Normal] MySQL 是一款开源、跨平台、高性能的关系型数据库管理系统（RDBMS），遵循客户 / 服务器架构，支持标准 SQL 语言，具有运行稳定、部署简便、响应速度快、安全性高等特点，被广泛应用于各类数据管理、Web 服务与数据分析系统中。
[P82|Normal] 在本 B 站 ACG 视频数据统计分析系统 中，MySQL 主要承担数据持久化存储与结构化管理的核心作用。系统通过 Python 爬虫获取 B 站 ACG 类视频的标题、UP 主、播放量、点赞数、投币数、收藏数、评论数等数据，这些数据具有字段明确、结构规整、统计需求强的特点，非常适合使用关系型数据库进行存储。MySQL 以数据表为基本单位，能够将多源、异构的 ACG 视频数据规范化存储，保证数据一致性、完整性与可查询性。MySQL 提供的 InnoDB 存储引擎支持事务安全（ACID）、行级锁与崩溃恢复机制，能够保证系统在批量写入、高频统计查询、数据更新等操作中的稳定性，适合数据量大、查询频繁的数据分析场景。
[P83|Heading 2] 2.4 Echarts
[P84|Normal] ECharts 是由百度开源的一款基于 JavaScript 的数据可视化图表库，提供丰富、流畅、交互性强的图表展示能力，支持折线图、柱状图、饼图、雷达图、词云图、热力图等几十种常见图表类型，兼容主流浏览器，支持移动端自适应，是国内应用最广泛的数据可视化工具之一。在本 B 站 ACG 视频数据统计分析系统 中，ECharts 承担前端数据可视化展示的核心功能。系统将从 MySQL 中获取的 ACG 视频数据（如播放量、点赞数、投币数、UP 主粉丝量、发布时间分布、热词统计等）通过 Django 后端传递至前端，由 ECharts 渲染为播放量趋势图、分区占比饼图、热度排行柱状图、视频类型统计、评论词云等可视化图表，实现数据的直观展示与深度分析，为用户提供清晰、高效的数据阅览体验。
[P85|Heading 2] 2.5 Requests
[P86|Normal] Requests 是基于 Python 语言开发的开源 HTTP 请求库，遵循 Apache2 开源协议，以简洁、易用、人性化的设计理念著称，是 Python 生态中最主流的网络请求工具，广泛应用于网络数据采集、接口调用、网页访问等场景。Requests 基于 urllib 进行高度封装，屏蔽了 HTTP 协议底层复杂的请求构造、响应处理、编码转换等操作，提供了统一且简洁的 API 接口，支持 GET、POST、PUT、DELETE 等全部 HTTP 请求方式，能够高效完成网页内容获取、表单提交、参数传递、会话维持等功能。
[P87|Normal] 该库具备完善的请求控制能力，支持请求头定制、超时设置、代理配置、Cookie 持久化、SSL 证书验证等高级特性，可有效模拟浏览器行为发起网络请求，提升数据采集过程中的稳定性与隐蔽性。同时，Requests 能够自动处理响应内容的编码格式，支持 JSON、XML、HTML 等多种数据格式的解析与读取，大幅降低了网络请求开发的复杂度。
[P88|Heading 2] 2.6 Django
[P89|Normal] Django 是一款基于 Python 编程语言的开源、高级、全栈式 Web 应用开发框架，遵循MTV（Model Template View）软件设计架构模式，以快速开发、代码整洁、组件化强、安全性高为核心理念，被广泛应用于企业级网站、数据管理平台、后台管理系统、数据分析系统等各类 Web 项目开发中。Django 框架采用松耦合、高内聚的模块化设计，将 Web 应用的数据模型、页面展示、业务逻辑进行明确分离，使系统结构更加清晰，便于开发、维护与扩展。其核心架构分为三个层次：Model 层负责数据结构定义与数据库交互；Template 层负责前端页面渲染与展示；View 层负责接收用户请求、执行业务逻辑并返回响应结果。这种分层设计有效提升了代码复用率，降低了模块间的依赖关系。

[P102|Normal] 用户管理：支持用户注册、登录（带验证机制），保障用户信息安全。
[P103|Normal] 数据查询：提供分页展示与多字段组合查询（如按标题、UP主、发布时间等），满足灵活检索需求。
[P104|Normal] 可视化大屏：提供不少于4个维度的可视化图表，展示热度趋势、分区分布、排行榜等关键指标。

[P109|Normal] 普通用户包括未登录的访客和已注册登录的用户，其功能权限略有差异。
[P110|Normal] （1）用户注册与登录
	系统需提供安全可靠的注册与登录机制。注册时，用户需填写必要信息（如用户名、密码、邮箱），并通过邮箱验证码或短信验证码验证身份；系统需校验用户名、邮箱的唯一性，密码需经加密存储。登录时，支持账号密码登录。登录成功后生成身份认证令牌（如JWT），用于后续请求的鉴权。用户可查看个人信息、修改密码。
[P111|Normal] （2）视频数据浏览与查询
	系统主页以列表形式展示爬取的ACG视频，支持分页显示（用户可自定义每页条数）。提供多字段组合查询功能：至少支持视频标题（模糊匹配）、UP主名称（精确或模糊匹配）、发布时间（区间选择）三个字段的组合查询，查询结果可按播放量、发布时间等排序。用户点击视频可进入详情页，查看完整信息及互动数据。

[P113|Normal] 系统管理员负责系统的整体维护，主要功能包括：

[P115|Normal] 爬虫管理：手动触发或配置爬虫任务，查看爬虫运行日志，调整爬虫策略。
[P116|Normal] 数据管理：对爬取数据进行校验、修复或删除，保证数据质量。
[P117|Normal] 系统监控：查看系统运行状态、用户活跃度、性能指标等。
[P118|Heading 2] 3.3非功能性需求
[P119|Normal] 非功能性需求关注系统的质量属性，直接影响用户体验和系统长期运行的稳定性。本系统在非功能性方面主要从数据与性能、安全性与易用性、可扩展性与可靠性三个维度进行规划。
[P120|Normal] 在数据与性能方面，系统初期预计存储约100条ACG视频数据，随着爬虫任务的每日执行，数据量以日均100条的速度增长。MySQL数据库需支持百万级数据量的高效存储与查询，同时配合Redis缓存热点数据（如排行榜、热门视频）以减轻数据库压力。性能上，页面加载和图表渲染应在2秒内完成，多字段组合查询响应时间不超过3秒；系统需支持至少50人同时在线访问，并发查询无明显延迟；爬虫单次采集任务应在10分钟内完成，并支持定时调度实现数据每日自动更新。
[P121|Normal] 在安全性与易用性方面，用户密码采用bcrypt等强加密算法存储，用户认证使用JWT令牌并设置合理有效期，前后端交互采用HTTPS协议防止数据泄露；在可扩展性与可靠性方面，系统采用模块化设计，各功能模块解耦，便于后期增加弹幕情感分析、个性化推荐等新功能；爬虫模块应具备良好的适配性，可扩展至其他视频平台的数据采集。

[P127|Heading 2] 4.1系统总体架构设计
[P128|Normal] 本系统采用前后端分离的分层架构，自底向上划分为数据采集层、数据存储层、业务逻辑层和用户界面层，各层之间通过RESTful API进行通信。整体架构如图
[P130|图标题] 图4.1 系统总体架构图
[P131|Normal] 用户界面层：基于Vue3框架和ElementPlus组件库构建单页应用，通过ECharts和DataV实现可视化大屏展示。界面层负责接收用户交互请求，调用后端API并将数据渲染到页面。
[P132|Normal] 业务逻辑层：采用Django + Django REST framework开发RESTful API服务，实现用户认证、数据查询、统计分析、收藏评论、算法处理等核心业务逻辑。同时使用Redis缓存热点数据（如排行榜），提升响应速度。

[P135|Normal] 系统采用前后端分离架构，前端通过Axios发送HTTP请求调用后端API，后端返回JSON格式数据。这种架构有利于前后端独立开发、部署和维护，同时便于后期扩展。

[P286|Normal] 创意方面，本项目的核心创意在于聚焦B站ACG垂直领域，针对二次元文化社区特有的“一键三连”（点赞、投币、收藏）、弹幕文化、分区标签等数据维度，设计了专门的数据采集模型和分析指标体系。与通用视频数据分析工具不同，本系统强调了ACG社区的热度评价方式（如投币和收藏权重高于普通点赞）这种垂直领域的深度整合，使得系统能够为ACG爱好者、UP主和运营者提供更具针对性的数据洞察。技术路线方面，系统采用前后端分离的分层架构，技术选型成熟且合理。前端使用Vue3 + ElementPlus + ECharts + DataV，实现了响应式单页应用和交互式可视化大屏；后端使用Django + Django REST framework + Redis，提供了高性能的RESTful API和缓存服务；数据采集层基于Scrapy框架，结合Requests、BeautifulSoup等库，实现了稳定、可调度的爬虫程序；数据存储采用MySQL + Redis的组合，兼顾了持久化和高速缓存需求。整个技术路线体现了大数据工程技术专业的数据采集、存储、处理、可视化的完整链条。文档撰写：毕业设计说明书正文包含需求分析、系统设计、实现细节、测试交付等完整章节，图表10余张。功能实现：完成了任务书要求的全部8项核心功能（数据库配置、用户注册登录、分页组合查询、可视化大屏），并额外实现了Redis缓存、JWT认证、图表联动等扩展功能。存在的不足：尽管系统基本达到了预期目标，但仍存在一些局限性。首先，爬虫数据源主要依赖B站公开API和页面解析，未接入WebSocket实时数据，导致互动数据（如弹幕）无法实时更新。另外，系统未实现移动端适配，在手机浏览器上显示效果欠佳。最后，由于B站页面结构可能变更，爬虫的长期稳定性需要持续维护。
[P287|Heading 2] 7.2 展望
[P288|Normal] 针对上述不足，结合大数据技术和ACG社区的发展趋势，本系统可以从以下几个方面进行进一步改进和拓展。
[P289|Normal] 引入实时数据流处理，目前系统采用定时爬虫（每日一次）的批处理模式，数据时效性为天级别。增加弹幕情感分析与内容理解，弹幕是B站ACG文化的核心组成部分，蕴含着丰富的用户情感和观点。后续可以基于自然语言处理（NLP）技术，对采集到的弹幕文本进行情感分类（正面、中性、负面）和关键词提取，分析用户对视频内容的评价倾向。此外，可利用视频标题、简介、标签等信息，通过LDA主题模型挖掘不同分区的热点话题，辅助内容创作。实现个性化推荐系统，基于用户的历史观看、收藏、评论行为，构建协同过滤或深度学习推荐模型（如YouTube的深度神经网络推荐算法），为用户推荐可能感兴趣的ACG视频。这需要增加用户行为日志采集模块（如记录用户点击、停留时长）。
